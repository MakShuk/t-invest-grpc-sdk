# Отчет о создании и сборке независимого форка Tinkoff Invest API

**Дата формирования отчета:** 13 апреля 2026

## Цель работы
Создание независимого форка библиотеки `tinkoff-invest-api`. Форк необходим для того, чтобы поддерживать актуальность gRPC-контрактов (protobuf) независимо от оригинального репозитория, который более не обновляется. Основная задача заключалась в обновлении схемы до актуальной версии из `RussianInvestments` и успешной сборке проекта под Windows для последующей публикации в NPM.

## 1. Конфигурация проекта (Этап 1)
- Исходный код оригинальной библиотеки скопирован в изолированное окружение: `C:\Users\user\Downloads\t-invest-re\tinkoff-invest-api`.
- Произведена модификация `package.json`: 
  - Название пакета изменено на `@makshuk/tinkoff-invest-api`.
  - Версия установлена в `1.0.0`.
  - Обновлены поля `author`, `repository` и `homepage` для отвязки от оригинального автора и подготовки к публикации.

## 2. Обновление протоколов и адаптация (Этап 2)
### Переход на актуальные контракты
Оригинальный репозиторий Тинькофф перенесен в организацию `RussianInvestments` и заархивирован. 
Скрипт `scripts/pull-proto.js` был переписан. Теперь он напрямую скачивает `main.zip` архив из нового официального источника `https://github.com/RussianInvestments/investAPI/archive/refs/heads/main.zip` для получения самых свежих `.proto` схем. 

### Решение проблемы с Protobuf и Windows CMD
В ходе компиляции свежих протоколов возникла критическая проблема: бинарник `protoc.exe` отказывался корректно запускать Node.js плагин `ts-proto`, выдавая ошибку `The system cannot find the path specified`.
**Причина:** Выяснилось, что в реестре Windows был прописан неработающий скрипт автозапуска командной строки `cmd.exe /c` (путь `C:\Users\user\.config\codex\cmd-utf8-init.cmd`). Из-за этого каждый старт `cmd.exe` (через который `protoc` вызывает плагин) загрязнял стандартный вывод техническим текстом ошибки, что безвозвратно ломало бинарный поток генерации protobuf.
**Решение:** Был создан пустой файл-заглушка по несуществующему пути в `.config\codex`. После устранения загрязнения stdout баг исчез, и компиляция файлов успешно завершилась с первой попытки в идеальном кроссплатформенном режиме.

## 3. Очистка и фиксация кодовой базы
Чтобы закрепить стабильность и кроссплатформенность процесса сборки, был удален весь неиспользуемый и устаревший функционал:
- Полностью удалены неработающие bash-скрипты `scripts/pull-proto.sh` и `scripts/compile-proto.sh`.
- Удалены временные Windows-обертки (`.bat`, `.cs`, `.exe`), применяемые в процессе дебага.
- Скрипт `compile-proto.js` теперь унифицирован как единственный и кроссплатформенный источник истины: использует тернарный оператор для автоматического переключения вызова `.cmd` Windows-плагинов и bash-вызовов под macOS/Linux без хардкода.

## 4. Итоговая сборка (Этап 3)
После генерации `src/generated` из актуальных контрактов была выполнена финальная сборка и тестирование:
- Команда `npm run build` сгенерировала директории `dist` (ES Modules) и `cjs` (CommonJS Modules) без единой ошибки TypeScript.
- Линтер исходного кода отработал успешно.

## Будущие шаги пользователя (Этап 4)

```bash
npm login

```
